模块全景 —— 它解决什么问题？

想象一下：你有几百个用户，每个用户同时进行多个 Agent 任务。每个任务都可能产生临时文件、对话记录、工具执行日志和最终产出物。 如果没有 Workspace 模块，这些文件会散落在各处，你无法知道哪个文件属于谁、哪个文件是哪个会话产生的、哪些文件可以安全删除。

Workspace 模块就是解决这个问题的。它借鉴了 MLflow "元数据与制品分离"的设计哲学：

  • 文件系统 —— 存放所有实际文件（对话记录、执行日志、产出物）
  • SQLite —— 存放元数据索引（用户、会话、产出物索引）
关键设计理念：扫描文件系统很慢。所以每次会话创建/文件生成时，同时在 SQLite 中写入一条索引记录。查询"某会话的所有产出物"只需一次 SQL，无需遍历目录。

目录结构 —— 每个文件的"家"

在深入 API 之前，先理解物理布局。下面是一个完整展开的用户工作区目录树：

users_workspace/ <-- WorkspaceManager.root └── <user_uuid>/ <-- _user_dir(uuid) ├── skills/ <-- 用户自定义技能（跨 session 持久化） ├── sessions/ │ └── <session_id>/ <-- _session_dir(uuid, sid) │ ├── session.yaml <-- 会话配置 & 状态 │ ├── conversation/ │ │ ├── messages.jsonl <-- 对话记录（JSONL） │ │ └── summary.md <-- 对话摘要 │ ├── execution/ │ │ ├── plan.yaml <-- 工具执行计划 │ │ └── logs/ <-- 执行日志目录 │ ├── workspace/ <-- nsjail 绑定挂载区（读写） │ │ ├── input/ <-- 输入文件 │ │ ├── scratch/ <-- Agent 临时工作区 │ │ └── output/ <-- 最终产出物 │ └── resources/ <-- 外部资源引用清单 │ └── resource_manifest.yaml ├── persistent/ <-- 用户长期资产（不被清理） └── user_profile.yaml <-- 用户偏好配置
重要：sessions/ 目录中的过期会话会被清理，但 persistent/ 和 skills/ 不受影响 —— 它们是用户的长期资产。

4.2 用户管理（1 个方法）

方法签名返回值用途

4.3 会话生命周期（4 个方法）

方法签名返回值用途
create_session(user_uuid, session_id=None, *, task_summary="", tags=None) Session 创建新会话。自动生成 session_id（时间戳+随机数避免碰撞）， 创建全部 7 个子目录，写入 session.yaml / plan.yaml / messages.jsonl / summary.md / resource_manifest.yaml。

list_sessions(user_uuid, limit=50, offset=0) list[Session] 分页列出用户的所有会话。按创建时间倒序。
session_id 的自动生成规则：
sess_{毫秒时间戳}_{8位随机hex} ，例如 sess_1717000000000_a1b2c3d4。
时间戳在前是为了让目录列表天然按时间有序；碰撞概率极低（同一毫秒内需有 2^32 次碰撞）。

方法签名返回值用途

内部路径工具 —— 所有路径的计算规则

模块中所有路径都是通过这两个私有方法计算的，它们是单一真相来源： 

def _user_dir(self, user_uuid: str) -> Path:
    return self.root / user_uuid

def _session_dir(self, user_uuid: str, session_id: str) -> Path:
    return self._user_dir(user_uuid) / "sessions" / session_id
为什么这是关键设计？
如果每个方法各自拼接路径（如 root + "/" + uuid + "/sessions/" + sid）， 一旦目录结构调整，你需要修改 N 处。现在只需修改这两个函数。

对应的常量定义在类顶部：

# 子目录名常量
DIR_SKILLS      = "skills"
DIR_SESSIONS    = "sessions"
DIR_WORKSPACE   = "workspace"
DIR_INPUT       = "input"
DIR_SCRATCH     = "scratch"
DIR_OUTPUT      = "output"
DIR_PERSISTENT  = "persistent"
# 等等...

WorkspaceDB 封装了所有 SQLite 操作。通常你不需要直接使用它 —— WorkspaceManager 已经为你调用了所有必要的方法。但理解它的接口有助于调试和扩展。

6.3 线程安全策略

每操作一个连接。
不使用共享连接或连接池。每个公开方法内部调用 _connect() 创建新连接， 操作完立即关闭。sqlite3 在 check_same_thread=False + WAL 模式下支持多线程并发读， 写入通过 5 秒超时重试解决偶发冲突。当前 MVP 单 Worker 场景下几乎不会触发。

端到端使用示例 —— 把一切串起来

下面是一个完整的生命周期，从初始化到清理：

# ============================================================
# 1. 初始化
# ============================================================
from pathlib import Path
from src.workspace import WorkspaceManager, SessionStatus

wm = WorkspaceManager(root=Path("users_workspace"))

# ============================================================
# 2. 确保用户存在（幂等）
# ============================================================
user = wm.ensure_user(
    user_uuid="alice-1234-5678",
    username="Alice",
)

# ============================================================
# 3. 创建会话
# ============================================================
session = wm.create_session(
    user_uuid="alice-1234-5678",
    task_summary="分析销售数据并生成月度报告",
    tags=["report", "monthly"],
)
print(f"会话创建: {session.session_id}")

# ============================================================
# 4. 获取 nsjail 挂载参数（传给沙箱）
# ============================================================
mounts = wm.get_nsjail_mounts(
    "alice-1234-5678", session.session_id
)
# => ["/path/.../workspace:/work", "/path/.../skills:/skills"]

# ============================================================
# 5. Agent 执行任务 ...（在 nsjail 沙箱中进行）
# ============================================================

# ============================================================
# 6. 注册产出物
# ============================================================
artifact = wm.register_artifact(
    session_id=session.session_id,
    file_path="output/monthly_report.pdf",
    file_type="application/pdf",
)

# ============================================================
# 7. 结束会话
# ============================================================
wm.end_session(
    session.session_id,
    status=SessionStatus.COMPLETED,
    task_summary="成功生成月度销售报告",
)

# ============================================================
# 8. 查询历史
# ============================================================
sessions = wm.list_sessions("alice-1234-5678")
for s in sessions:
    print(f"{s.session_id}: {s.status} - {s.task_summary}")
    artifacts = wm.list_artifacts(s.session_id)
    for a in artifacts:
        print(f"  产出: {a.file_path} ({a.file_size} bytes)")

# ============================================================
# 9. 定期清理（可设为 cron job）
# ============================================================
cleaned = wm.cleanup_expired_sessions(max_age_days=30)
print(f"清理了 {cleaned} 个过期会话")

设计决策 —— 为什么这样设计？

为什么用 SQLite 而不是纯文件系统？

如果纯靠文件系统，列举"Alice 的所有会话"需要遍历目录树，列举"某会话的所有产出物"需要扫描 output/ 目录。当会话数量达到数百个时，这些操作会变得很慢。

SQLite 的 B-Tree 索引让这些查询始终保持 O(log n)，而且它是嵌入式数据库，不需要额外部署服务。

为什么使用每个操作新建连接，而不是连接池？

Python 的 sqlite3 模块不支持跨线程共享连接（会抛 ProgrammingError）。虽然在 WAL 模式下可以通过 check_same_thread=False 绕过，但更好的做法是每操作一个连接 —— 它足够轻量（文件级别），且避免了并发写入时的锁竞争问题。

当前 MVP 是单 Worker 模式，性能不是瓶颈。如果未来需要更高并发，应该切换到 PostgreSQL 而非优化 SQLite 连接管理。

YAML 为什么用原子写入（.tmp + os.replace）？

防止写入过程中进程崩溃导致文件损坏。写入流程：

1. yaml.dump(data) → content (内存中的字符串)
2. file_store.write(path + ".tmp", content) → 写入临时文件
3. os.replace(tmp_path, target_path) → 原子 rename

os.replace 是 POSIX 原子操作，保证不会出现"文件存在但内容不完整"的中间状态。

session_id 为什么以时间戳开头？

两个好处：

  1. 文件系统天然有序 —— 在文件管理器中默认按名称排序时，会话自动按创建时间排列
  2. 碰撞概率极低 —— 毫秒时间戳 + 8位随机hex = 2^32 种可能/毫秒，实际场景下几乎不可能碰撞
