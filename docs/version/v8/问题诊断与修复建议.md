## 问题 1：Hindsight API 不可用导致每次消息回复耗时 60-70 秒

### 日志证据

```
19:44:04  Received private message from 2109279314: 你说的对
19:44:35  WARNING Hindsight API timeout: POST .../memories/recall    ← 等满 30s
19:45:05  WARNING Hindsight API timeout: POST .../memories/recall    ← 又等满 30s
19:45:13  Sent message to NapCat clients                              ← LLM 实际只用了 ~8s

19:46:13  Received private message from 2109279314: 是的 你有什么建议吗
19:46:43  WARNING Hindsight API timeout: POST .../memories/recall    ← 等满 30s
19:47:16  WARNING Hindsight API timeout: POST .../memories/recall    ← 又等满 30s
19:47:25  Sent message to NapCat clients                              ← LLM 实际只用了 ~9s
```

### 根因

在 `src/harness/runner.py:151`，`recall_memories()` 在 agent 主循环内**同步阻塞**执行，LLM 调用必须等待 recall 完成（或超时）才能开始。Hindsight 服务（`localhost:8001`）不可用/响应极慢时，每次 HTTP 调用等满 30 秒超时。

**延迟归因**：

```
每条消息 ~70s 总延迟 =
    第一次 recall 超时  30s  ← Hindsight 不可用
  + 第二次 recall 超时  30s  ← 同一次查询打了两次（见问题 2）
  + LLM API 调用       ~8s  ← 合理
  + 其他               <1s
```

即 **85% 的延迟来自 Hindsight 超时等待**。

### 推荐修复

**方案 A（推荐 — 快速失败 + 降级）**：
- 将 recall 超时从 30s 降为 3s（recall 是实时检索，不应超过 3 秒）
- 保留 retain 超时为 10s（retain 是写入操作，可以稍长）
- 超时后直接降级为空记忆，不阻塞主流程

**方案 B（异步化）**：
- 将 recall 改为 `asyncio.create_task()` 异步发起，同时开始构建 LLM 请求
- 用 `asyncio.wait_for(memories_task, timeout=3.0)` 在 LLM 调用前获取结果
- 如果 3 秒内未返回，直接用空记忆发送 LLM 请求

**方案 C（治本）**：
- 排查 Hindsight 服务为什么不可用（`localhost:8001` 是否正常运行、pgvector 是否健康、是否有资源瓶颈）
- 给 Hindsight 加上健康检查端点，在启动时验证连通性

### 涉及文件

| 文件 | 行号 | 说明 |
|------|------|------|
| `src/harness/runner.py` | 151 | recall 在主循环同步阻塞点 |
| `src/session/store.py` | 213-228 | recall_memories 实现 |
| `src/memory/hindsight_client.py` | 73 | timeout 默认值 30.0 |
| `src/orchestration/config.py` | 204 | HindsightConfig.timeout 配置 |

## 问题 2：recall_memories() 重复调用 recall() 两次

### 日志证据

两次连续的 recall 超时（19:44:35 和 19:45:05），间隔恰好 30s，说明同一轮消息处理中发起了两次独立的 recall HTTP POST。

### 根因

`src/session/store.py:223-228`：

```python
async def recall_memories(self, query, account_id, session_id, top_n=5):
    items = await self._hindsight.recall(query, ...)        # 第 1 次 HTTP POST
    formatted = await self._hindsight.recall_formatted(...)   # 第 2 次 HTTP POST
    return items, formatted
```

而 `recall_formatted()` (`src/memory/hindsight_client.py:331`) 内部**又调用了一次 `self.recall()`**：

```python
async def recall_formatted(self, query, user_id, session_id, top_n=5):
    items = await self.recall(query, user_id, session_id, top_n)  # ← 完全相同的 HTTP 请求
    if not items:
        return ""
    # ... 格式化为文本
```

等于同一条 query 对 Hindsight 发起了两次完全相同的 HTTP POST 请求。在 Hindsight 正常时浪费 1 倍网络延迟，在 Hindsight 不可用时浪费 30 秒超时。

### 推荐修复

**方案 A（最简单）**：让 `recall_memories()` 只调用一次 `recall()`，然后本地格式化：

```python
async def recall_memories(self, query, account_id, session_id, top_n=5):
    if not self._hindsight:
        return [], ""
    try:
        items = await self._hindsight.recall(query, account_id, session_id, top_n)
        if not items:
            return [], ""
        # 本地格式化，不再发第二次 HTTP 请求
        formatted = HindsightClient.format_memories(items)
        return items, formatted
    except Exception as e:
        logger.warning("DEGRADATION: Hindsight recall failed (%s)", e)
        return [], ""
```

**方案 B**：让 `recall_formatted()` 接受可选的 `items` 参数，当外部已调用过 `recall()` 时直接复用结果，不发 HTTP 请求。

### 涉及文件

| 文件 | 行号 | 说明 |
|------|------|------|
| `src/session/store.py` | 224-227 | 连续两次调用 recall |
| `src/memory/hindsight_client.py` | 313-345 | recall_formatted 内部重复调用 recall |

## 问题 3：HindsightClient 无重试/熔断机制

### 日志证据

全部 6 次 Hindsight 调用（2 条消息 × 3 次）全部超时，没有一次成功。每次都等满 30 秒后放弃。

### 根因

`src/memory/hindsight_client.py:96-123` 的 `_request()` 方法：

```python
async def _request(self, method, path, json_data=None):
    try:
        async with httpx.AsyncClient(timeout=self.timeout) as client:
            ...
    except httpx.TimeoutException:
        logger.warning("Hindsight API timeout: %s %s", method, path)
        return None    # ← 直接返回，无重试
    except httpx.HTTPStatusError as e:
        logger.warning("Hindsight API error: %s %s → %s", method, path, e.response.status_code)
        return None    # ← 无重试
    except Exception as e:
        logger.warning("Hindsight API unreachable: %s %s → %s", method, path, e)
        return None    # ← 无重试
```

缺失的能力：
1. **重试**：网络抖动导致的一过性超时没有机会恢复
2. **指数退避**：重试之间没有等待间隔
3. **熔断器**：连续失败后不会自动停止尝试，每次都浪费 30s
4. **健康检查**：没有定期探测 Hindsight 是否已恢复

### 推荐修复

**短期（加熔断器）**：

```python
class HindsightClient:
    def __init__(self, ...):
        self._failure_count = 0
        self._circuit_open_until = 0

    async def _request(self, method, path, json_data=None):
        # 熔断：连续失败 5 次后，30 秒内直接返回 None，不发起 HTTP 请求
        if self._failure_count >= 5:
            if time.monotonic() < self._circuit_open_until:
                return None
            self._failure_count = 0  # 试探性恢复

        try:
            ...
            self._failure_count = 0  # 成功则清零
            return result
        except httpx.TimeoutException:
            self._failure_count += 1
            if self._failure_count >= 5:
                self._circuit_open_until = time.monotonic() + 30
            logger.warning("...")
            return None
```

**长期**：引入 `tenacity` 库实现重试 + 指数退避 + 熔断：

```python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(2), wait=wait_exponential(multiplier=0.5, max=3))
async def _request(self, method, path, json_data=None):
    ...
```

| 文件 | 行号 | 说明 |
|------|------|------|
| `src/memory/hindsight_client.py` | 96-123 | _request 无重试/熔断 |

## 问题 4：每次收到消息都重建 workspace/sandbox（SQLite UNIQUE 冲突）

### 日志证据

```
19:44:04  WARNING Workspace/sandbox setup failed for session-xxx:
         UNIQUE constraint failed: sessions.session_id
19:44:04  Started new session ...          ← 第一条消息：workspace 失败但 Temporal 成功

19:46:13  WARNING Workspace/sandbox setup failed for session-xxx:
         UNIQUE constraint failed: sessions.session_id
19:46:13  Signaled existing session ...    ← 第二条消息：同样失败
```

**每条消息都出现此 WARNING**。

### 根因

**调用链**：

1. `worker.py:271`：`session_id = f"session-{account_id}"` —— 确定性的，同一用户每次消息生成相同 ID
2. `worker.py:284-288`：每次消息都调用 `workspace_manager.create_session()` —— 无条件执行
3. `workspace/manager.py:238`：最终执行 `db.insert_session(session)`
4. `workspace/db.py:164`：`INSERT INTO sessions (session_id, ...) VALUES (?, ...)` —— 原始 INSERT，无去重
5. `workspace/db.py:46`：`session_id TEXT PRIMARY KEY` —— SQLite 主键隐式 UNIQUE

**时序分析**：

```
第一条消息:  INSERT session → 成功（DB 为空） → Temporal start_workflow → 成功
第二条消息:  INSERT session → 失败（主键冲突）→ 被 try/except 吞掉
            → sandbox 创建（新 UUID，不冲突，但旧 sandbox 泄漏）
            → Temporal signal → 成功
第 N 条消息: 同上，每次都在 SQLite 报 UNIQUE 冲突
```

**附带问题**：sandbox 创建在 try 块内，workspace 失败不阻止 sandbox 创建（sandbox UUID 每次不同所以不报错），导致每次消息**泄露一个新 sandbox 实例**。

### 推荐修复

**方案 A（最小改动）**：将 SQLite INSERT 改为幂等操作：

```python
# workspace/db.py insert_session()
conn.execute(
    """INSERT OR IGNORE INTO sessions (...) VALUES (...)""",
    ...
)
```

**方案 B（更彻底）**：在 `worker.py` 中先检查 session 是否已存在，已存在则跳过整个 workspace/sandbox 设置块：

```python
async def handle_message(message):
    session_id = f"session-{account_id}"
    existing = await session_store.get_session(session_id)
    if existing is None:
        # 仅首次消息执行 workspace/sandbox 创建
        try:
            workspace_manager.ensure_user(account_id)
            workspace_manager.create_session(...)
            sandbox_manager.create_session_sandbox(...)
        except Exception as e:
            logger.warning(...)
    # ... Temporal start/signal
```

### 涉及文件

| 文件 | 行号 | 说明 |
|------|------|------|
| `src/orchestration/worker.py` | 271 | session_id 确定性生成 |
| `src/orchestration/worker.py` | 284-297 | 无条件创建 workspace/sandbox |
| `src/workspace/db.py` | 46 | PRIMARY KEY 约束 |
| `src/workspace/db.py` | 149-174 | insert_session 无幂等处理 |

## 问题 5：workspace 创建与 Temporal workflow 启动非原子操作

### 日志证据

```
19:44:04  WARNING Workspace/sandbox setup failed ← workspace 失败
19:44:04  Started new session ...                ← Temporal 成功
```

两者在同一个消息处理流程中独立执行，缺乏事务性保证。

### 根因

`worker.py:282-313` 的结构：

```python
# 块 1：workspace/sandbox（独立 try/except）
try:
    workspace_manager.create_session(...)
    sandbox_manager.create_session_sandbox(...)
except Exception as e:
    logger.warning("Workspace/sandbox setup failed: %s", e)
    # ← 异常被吞掉，不影响后续流程

# 块 2：Temporal workflow（独立 try/except）
try:
    await client.start_workflow(...)
except WorkflowAlreadyStartedError:
    await handle.signal(...)
```

两个块互不感知对方的状态。可能导致：
- **workspace 失败 + Temporal 成功**：Temporal workflow 启动后，agent 尝试访问不存在的 workspace 目录（当前被后续的 `_ensure_session` 掩盖）
- **workspace 成功 + Temporal 失败**：SQLite 中有 session 记录，但 Temporal 没有对应的工作流

### 推荐修复

将 workspace/sandbox 创建移到 Temporal workflow 内部（作为 activity 的第一步），而不是在 worker 的消息入口处执行。这样：
- workspace 创建和 workflow 执行在同一生命周期内
- 首次消息和后续消息的代码路径统一（都在 workflow 内处理）
- 避免 worker 层承担业务逻辑

如果保持当前架构不变，至少应该：
```python
try:
    workspace_manager.create_session(...)
    sandbox_manager.create_session_sandbox(...)
except Exception as e:
    logger.error("Workspace setup failed, aborting: %s", e)
    return  # ← 不继续启动 Temporal workflow
```

### 涉及文件

| 文件 | 行号 | 说明 |
|------|------|------|
| `src/orchestration/worker.py` | 282-313 | 两个独立 try/except 块 |

## 问题 6：Hindsight 超时配置为单一全局值

### 现象

所有 Hindsight 操作共用一个 30 秒超时，无法区分：
- **recall**（检索）：应该快（<3s），超时应该短
- **retain**（写入）：涉及 embedding 生成 + pgvector 写入，可以稍长（10-15s）
- **reflect**（深度推理）：服务端计算密集型，可能需要 30-60s
- **ensure_bank**（创建 bank）：一次性操作，应该快

### 根因

`src/memory/hindsight_client.py:73`：

```python
class HindsightClient:
    def __init__(self, ..., timeout: float = 30.0):
        self.timeout = timeout
```

`src/orchestration/config.py:204`：

```python
@dataclass
class HindsightConfig:
    timeout: float = 30.0
```

只有一个 `timeout` 参数，所有 `_post()`, `_get()`, `_put()` 共用。

### 推荐修复

为不同操作配置不同超时：

```python
@dataclass
class HindsightConfig:
    recall_timeout: float = 5.0     # 检索应该快
    retain_timeout: float = 15.0    # 写入允许慢
    reflect_timeout: float = 60.0   # 深度推理允许更慢
    bank_timeout: float = 10.0      # 创建 bank

class HindsightClient:
    def __init__(self, config: HindsightConfig):
        self._timeouts = {
            "recall": config.recall_timeout,
            "retain": config.retain_timeout,
            "reflect": config.reflect_timeout,
            "bank": config.bank_timeout,
        }

    async def _post(self, path, json_data, op: str):
        timeout = self._timeouts.get(op, 30.0)
        async with httpx.AsyncClient(timeout=timeout) as client:
            ...
```

| 文件 | 行号 | 说明 |
|------|------|------|
| `src/memory/hindsight_client.py` | 73, 104 | 单一超时参数 |
| `src/orchestration/config.py` | 204 | 单一配置字段 |

## 问题 7：Temporal Activity retry=1（等于不重试）

### 现象

Activity 如果因临时故障失败（如 Hindsight 短暂不可用后立即恢复、Redis 网络抖动），不会重试，直接返回空结果给用户。

### 根因

`src/orchestration/workflow.py:104-108`：

```python
result = await workflow.execute_activity(
    "process_turn_activity",
    args=[user_message],
    start_to_close_timeout=timedelta(seconds=120),
    retry_policy=RetryPolicy(
        maximum_attempts=1,           # ← 不重试
        initial_interval=timedelta(seconds=1),   # 配置了但无效
        maximum_interval=timedelta(seconds=60),  # 配置了但无效
    ),
)
```

`maximum_attempts=1` 意味着只执行一次，失败即终止。`initial_interval` 和 `maximum_interval` 已定义但因为 `max_attempts=1` 而不生效。

### 推荐修复

```python
retry_policy=RetryPolicy(
    maximum_attempts=3,
    initial_interval=timedelta(seconds=1),
    maximum_interval=timedelta(seconds=10),
    non_retryable_error_types=["NonRetryableError"],  # 对于确定性失败不重试
),
```

同时需要在 activity 中区分可重试异常（超时、连接拒绝）和不可重试异常（参数错误、断言失败），对后者抛出 `NonRetryableError`。

| 文件 | 行号 | 说明 |
|------|------|------|
| `src/orchestration/workflow.py` | 104-108 | retry_policy 配置 |

## 问题 8：会话 idle 超时仅 1 分钟

### 日志证据

```
19:47:55  最后一次 Hindsight retain 超时完成
19:48:57  Session session-xxx idle for 1 minute, auto-closing
```

### 根因

`src/orchestration/workflow.py` 中 workflow 的 idle timeout 被配置为 1 分钟。`19:48:57 - 19:47:55 ≈ 62s`，从最后一个 activity 完成到触发关闭约 1 分钟。

对于真实聊天场景，1 分钟太短：
- 用户需要时间阅读 AI 的回复（尤其当回复较长时）
- 用户需要时间思考如何回复
- 用户可能在打字中途被打断

### 推荐修复

将 idle timeout 提升到 5-10 分钟：

```python
# workflow.py
await workflow.wait_condition(
    lambda: bool(new_message_received),
    timeout=timedelta(minutes=10),  # 从 1 分钟改为 10 分钟
)
```

或使 idle timeout 可配置：

```python
# config
@dataclass
class TemporalConfig:
    session_idle_timeout_minutes: int = 10
```

## 问题 9：LLM 调用使用 stream=False

### 现象

用户需要等待 LLM 生成完整回复后才能看到任何内容。在当前的 ~8s LLM 推理时间内，用户体验是"发送消息后等待 70 秒，然后突然出现完整回复"。

### 根因

`src/harness/runner.py:169`：

```python
response = await self._model.generate(
    model_selector="chat",
    messages=context,
    tools=tools,
    stream=False,  # ← 非流式
)
```

而 `ModelClient` 已经实现了完整的流式解析能力（`_parse_anthropic_stream` 在 `model_client.py:280`，`_parse_openai_stream` 在 `model_client.py:350`），只是没有被调用。

### 推荐修复

**短期**（对延迟改善最明显，结合问题 1 一起修复后效果更佳）：

将 runner.py 改为 `stream=True`，并在收到第一个 token 后通过 NapCat 逐条发送。对于工具调用场景，可以先缓冲完整响应再判断（因为 tool_calls 需要完整 JSON）。

```python
response = await self._model.generate(
    model_selector="chat",
    messages=context,
    tools=tools,
```

**注意**：流式输出与工具调用有交互复杂性——如果模型返回 tool_calls，需要先收集完整响应再执行工具。可以这样处理：
- 如果第一个 chunk 就表明是 tool_calls，回退到非流式或缓冲完整响应
- 如果是普通文本，逐 token 推送给用户

## 问题 10：retain_memories 阻塞 activity 完成

### 现象

虽然用户已经收到回复，但 `process_turn_activity` 在 retain 完成前不会返回。这导致：
- Temporal workflow 在 retain 超时期间（30s）无法接收新的 signal
- 如果用户恰好在 retain 期间发送下一条消息，signal 会在 Temporal 服务端排队

### 日志证据

两个消息周期之间的时间线：
```
第一条: 19:45:13 响应发送 → 19:45:43 retain 超时 → activity 完成
第二条: 19:46:13 收到新消息（恰好是 activity 完成 +30s 后）
```

虽然第二条消息在 19:46:13 到达，但第一条消息的 activity 直到 19:45:43 才真正完成。如果用户在 19:45:20 发了消息，signal 需要排队等待 23 秒。

### 根因

`src/harness/runner.py:230-233`：

```python
await self._send_response(final_content, user_message)       # 响应已发出
...
await self._session.retain_memories(turn_events, ...)         # ← 阻塞 activity 完成
```

### 推荐修复

将 retain 改为真正的 fire-and-forget：

```python
await self._send_response(final_content, user_message)
# 不 await，让 retain 在后台执行
asyncio.create_task(self._safe_retain(turn_events, account_id, session_id))
```

或在 Temporal 层面将 retain 拆分到独立的 activity，与主 activity 并行执行：

```python
# workflow.py
await asyncio.gather(
    workflow.execute_activity("process_turn_activity", ...),
    workflow.execute_activity("retain_activity", ...),
)
```

| 文件 | 行号 | 说明 |
|------|------|------|
| `src/harness/runner.py` | 233 | retain_memories 被 await |
| `src/orchestration/workflow.py` | 流程编排 | 可拆分 activity |

## 问题 11：工作区/上下文同步文件 I/O 阻塞事件循环

### 现象

在 asyncio 事件循环中执行同步文件 I/O，高并发时会导致事件循环阻塞，所有协程被暂停。

### 根因

以下位置在事件循环线程中执行了同步阻塞调用：

| 位置 | 阻塞操作 | 风险 |
|------|----------|------|
| `worker.py:284-285` | `workspace_manager.ensure_user()` — `os.makedirs` | 首次创建用户目录时可能触发磁盘 I/O |
| `worker.py:286-290` | `workspace_manager.create_session()` — SQLite `INSERT` + `COMMIT` | SQLite 写入是同步的，数据库锁等待时阻塞 |
| `worker.py:291-295` | `sandbox_manager.create_session_sandbox()` — `os.path.isfile` | 检查 nsjail 二进制，文件系统访问 |
| `context_builder.py:128-201` | 读取 `.hermes.md`, `CLAUDE.md`, `SOUL.md` | 每次 agent turn 可能读取多个配置文件 |

### 推荐修复

- 文件 I/O 使用 `asyncio.to_thread()` 或 `loop.run_in_executor()` 移动到线程池
- SQLite 使用 `aiosqlite` 替代同步 `sqlite3`
- 配置文件在启动时预加载到内存，避免热路径上的文件读取

```python
# 示例：将同步 DB 操作移到线程池
await asyncio.to_thread(self.db.insert_session, session)

# 示例：启动时预加载 context 文件
class ContextBuilder:
    def __init__(self, ...):
        self._hermes_md = self._load_file("....hermes.md")  # 启动时加载
```

### 涉及文件

| 文件 | 行号 | 说明 |
|------|------|------|
| `src/orchestration/worker.py` | 284-295 | workspace/sandbox 同步创建 |
| `src/harness/context_builder.py` | 128-201 | 同步文件读取 |
| `src/workspace/db.py` | 149-174 | 同步 SQLite 操作 |
| `src/workspace/manager.py` | 146-240 | create_session 同步方法 |
